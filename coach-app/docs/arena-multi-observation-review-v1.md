# Arena multi-observation — review list (design V1)

Документ описывает **UI/UX и форму данных** для списка проверки перед сохранением нескольких наблюдений из одной голосовой фразы.  
**Не внедрено** в live runtime, не меняет существующий API и текущий confirm-flow.

**Связанные dev-слои:** [`split-arena-observations-v1.ts`](../dev/split-arena-observations-v1.ts), [`arena-multi-intent-adapter-v1.ts`](../dev/arena-multi-intent-adapter-v1.ts), [`arena-candidate-resolution-v1.ts`](../dev/arena-candidate-resolution-v1.ts).

---

## 1. Назначение review list

После разбиения транскрипта и resolution (`keep` / `merge` / `ambiguous`) тренеру нужен **один экран проверки**: что уйдёт в учёт как отдельные наблюдения, что объединить, что уточнить или отбросить.  
Это **post-session workflow**, а не часть живого голосового цикла.

---

## 2. Структура списка (три секции)

### SECTION A — READY TO KEEP

- Элементы, для которых resolution вернул **`keep`**.
- Смысл: система считает сегмент самодостаточным наблюдением; тренер может **подтвердить как есть** или при необходимости открыть деталь (вне scope V1 design).

**Что видит тренер (на элемент):**

| Поле | Описание |
|------|----------|
| **Segment text** | Текст сегмента (как после split). |
| **Subject** | Игрок (имя/номер) / команда / unknown — из `intent.kind` + отображаемая подпись. |
| **Sentiment** | positive / neutral / negative (из `intent`, для `unknown` — лексический слой аудита или «—» до согласования). |
| **Reason** | Кратко: например «отдельное наблюдение без конфликтующих правил V1» (из resolution `reason` или локализованная копия). |
| **Suggested action** | «Подтвердить как наблюдение» (primary). |

---

### SECTION B — NEEDS MERGE REVIEW

- Элементы, для которых resolution вернул **`merge`** (`sourceIndexes`, `mergedIntoIndex`).
- Смысл: два соседних сегмента относятся к **одному игроку**; нужно решить, сохранять **одно** наблюдение или **два**.

**Что видит тренер:**

| Поле | Описание |
|------|----------|
| **Segment text** | Оба фрагмента (сверху вниз или в одной карточке с разделителем «+»), с подсветкой общего `playerId`. |
| **Subject** | Один игрок (общий). |
| **Sentiment** | Опционально: если в сегментах разный тон — показать оба или итог «смешанный» (design: badge «разный тон»). |
| **Reason** | Текст из resolution, напр. дубликат одного игрока в двух сегментах. |
| **Suggested action** | «Объединить в одно наблюдение» (рекомендуемое) / «Оставить как два отдельных». |

---

### SECTION C — NEEDS ATTENTION

- Элементы **`ambiguous`**: shared-pair, продолжение team-контраста, и любые будущие эвристики «нужен человек».
- Смысл: автоматика **не утверждает** финальный смысл без решения тренера.

**Что видит тренер:**

| Поле | Описание |
|------|----------|
| **Segment text** | Текст сегмента (или пары, если ambiguous выдан на оба в shared-pair — две строки в одной карточке). |
| **Subject** | Как распознано сейчас (может быть неверно или incomplete). |
| **Sentiment** | Если применимо. |
| **Reason** | Напр. «общая оценка пары» / «продолжение оценки команды». |
| **Suggested action** | См. раздел «Действия» ниже — набор вторичных кнопок/меню. |

**Визуально:** секция C визуально отделена от A (например нейтральный акцент, иконка «внимание» без красной тревоги).

## 3. Действия тренера (design-level, без handlers)

### KEEP

- **Confirm as is** — зафиксировать наблюдение в том виде, как показано (subject + sentiment + текст).

### MERGE

- **Merge into previous** — одно итоговое наблюдение по объединённому тексту (или по «якорному» сегменту + пометка «дополнено»); детали объединения текста — продуктовое решение позже.
- **Keep separate** — два независимых наблюдения несмотря на совпадение `playerId`.

### AMBIGUOUS

- **Assign player** — выбрать игрока из ростера (или уточнить номер).
- **Convert to team** — трактовать как `create_team_observation` (если текст про группу/команду).
- **Keep as note** — сохранить как текстовую заметку / unknown без привязки к игроку (политика продукта).
- **Discard** — не создавать наблюдение по этому сегменту.

---

## 4. Визуальный приоритет и тон

- **Контекст:** экран открывается **после тренировки** или после пакета голосовых вводов, не во время активного упражнения на льду.
- **Тон:** спокойный, без таймеров и «срочно ответь»; нет имитации чата.
- **Не CRM-таблица:** не плотная сетка колонок; **карточки** или **секционированный список** с короткими блоками текста.
- **Иерархия:** сверху краткий summary («N готово, M на проверку»), затем **A → B → C** с явными заголовками секций.
- **Разделение:** «готово» (A) визуально легче; «нужно решение» (B, C) — чуть выше контраста фона или левый акцент-бордер.

## 5. Минимальная структура данных для UI

Базовый тип элемента списка (можно хранить в состоянии экрана или собрать из `ArenaResolutionResult` + исходных `ArenaIntentCandidate`):

```typescript
type ArenaReviewListItem = {
  id: string;
  resolutionKind: "keep" | "merge" | "ambiguous";
  segmentText: string;
  /** intent.kind сегмента-якоря или агрегата */
  intentKind: "create_player_observation" | "create_team_observation" | "unknown";
  playerId?: string | null;
  playerLabel?: string | null;
  sentiment?: "positive" | "neutral" | "negative";
  reason?: string;
  /** Для merge: индексы кандидатов в исходном массиве */
  sourceIndexes?: number[];
  mergedIntoIndex?: number;
  /** Для карточек с двумя сегментами (merge / dual ambiguous) */
  secondarySegmentText?: string;
  /** Порядок внутри секции */
  order: number;
};
```

**Дополнительные поля (по желанию, без изменения production API):**

- `originalTranscript: string` — полная фраза для контекста в свёрнутом блоке «Показать оригинал».
- `segmentIndex: number` — индекс сегмента после split (отладка / аналитика).
- `confidence?: number` — для player observation, если показывать тренеру «уверенность».
- `resolutionRuleId?: string` — стабильный ключ правила (duplicate / sharedPair / teamContinuation) для телеметрии.

**Маппинг (концептуально):**

- Один `keep` → один `ArenaReviewListItem`, `resolutionKind: "keep"`.
- Один `merge` → один item в секции B, `segmentText` + `secondarySegmentText` из двух кандидатов.
- Каждый `ambiguous` → один item в секции C; при двух ambiguous подряд (shared-pair) можно **одна карточка** с двумя текстами и `resolutionKind: "ambiguous"` + расширенный UI (два sub-rows).

## 6. Flow notes (куда встраивать)

- **Когда:** после завершения сессии / блока ввода, когда уже есть накопленные «сырые» multi-candidate результаты (или по кнопке «Разобрать фразы» в dev → позже в проде).
- **Где:** отдельный экран или шаг мастера «Проверка наблюдений» в том же приложении тренера, **не** внутри live voice sheet.
- **Почему не live-step:** разбиение и resolution могут ошибаться; в live режиме это ломает ритм и повышает когнитивную нагрузку. Post-session даёт время, меньше стресса и лучше качество данных.
- **Связь с текущим confirm-flow:** новый экран **не заменяет** существующий single-intent confirm; это **параллельный** сценарий для multi-observation, пока не подключён к runtime.

## 7. Следующий шаг после design doc (implementation)

1. **Чистая функция** `resolutionResultToReviewItems(...)` в `dev/` (или `lib/arena/` с пометкой experimental), без импорта в voice hook.  
2. **Статический или Storybook** прототип экрана с мок-данными из audit-сценариев multi-intent + resolution.  
3. Согласование с продуктом: тексты `reason` / `suggested action` на русском и правила для **merge** текста и **sentiment** при объединении.
