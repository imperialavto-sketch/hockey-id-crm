# Hockey ID — Reality Check Report

**Дата:** 2026-03-10  
**Цель:** Честная оценка готовности системы для реальных пользователей (родителей и школ).

---

## 1. Что уже реально можно использовать родителю сегодня

**Очень мало.**

- **Вход/регистрация по email (только если бэкенд — hockey-server)**  
  В parent-app используется `loginRequest`/`registerRequest` → `POST /api/auth/login` и `/api/auth/register`.  
  **hockey-server** (`hockey-server/controllers/authController.js`) реализует bcrypt + JWT.  
  Токен хранится в SecureStore и отправляется как `Authorization: Bearer`.  
  **Используемо только если `API_BASE_URL` указывает на hockey-server** (`parent-app/config/api.ts` → `http://192.168.1.45:3000`).  
  Если указать Next.js — логин не сработает (Next.js отдаёт `{ user, role }` и cookie, без JWT).

- **Просмотр данных «своего» ребёнка (с критическим риском)**  
  Родитель может открыть профиль игрока, расписание и т.д., но **hockey-server не фильтрует по parent**.  
  `GET /api/players` возвращает всех игроков; `GET /api/players/:id` не проверяет `parentId`.  
  То есть **интерфейс работает, но бэкенд позволяет любому залогиненному родителю видеть и открывать чужих детей** (IDOR).  
  «По-настоящему пригодно» станет только после исправления этой проверки.

- **Дашборд (для школы, не для родителей)**  
  Next.js CRM (`src/app/(dashboard)/`) реально используется персоналом: дашборд, игроки, команды, расписание, финансы, тренеры, маркетплейс, аналитика, настройки, лента, коммуникации.  
  Всё опирается на Prisma и `requirePermission`/data-scope по школе/команде.  
  Это **не** родительский сценарий — это та часть, которая «по-настоящему пригодна» для одной школы сегодня (при демо-авторизации; см. ниже).

**Итог:** Для **родителя** единственное, что работает end-to-end с реальной авторизацией — **вход/регистрация по email против hockey-server**. Всё остальное либо сломано (не тот бэкенд), либо небезопасно (нет фильтра по родителю), либо не реализовано (брони, оплаты, лента и т.д.).

- **Подписка**

- **Развитие игрока / паспорт / AI-отчёт**

  Используются mock-уведомления.

- **Логин дашборда (CRM)**  
  `src/app/api/auth/login/route.ts`: демо-пользователи + фиксированный пароль «admin123»; сессия в cookie `hockey-crm-session` (base64 JSON). Нет нормального «забыл пароль» и продакшн-авторизации.

- **Вход по телефону (родитель)**  
  Request-code — заглушка (нет SMS); verify в dev принимает код `1234`; JWT нет, сессия только в AsyncStorage — только демо.

  При регистрации письмо не отправляется и не проверяется.

  Нет app-level ErrorBoundary в parent-app и дашборде; только стандартные от фреймворка.

## 4. Топ-10 причин, почему система не выдержит, если завтра начнёт пользоваться реальная школа

1. **Родитель видит чужих детей (IDOR)**  
   В hockey-server `GET /api/players` и `GET /api/players/:id` не фильтруют по `parentId`. Любой залогиненный родитель может получить список и открыть любого игрока. **Критическая утечка данных.**

2. **Один базовый URL: либо логин, либо функции**  
   В `parent-app/config/api.ts` один `API_BASE_URL`.  
   Если это hockey-server: логин работает, но лента, чат, уведомления, брони, маркетплейс, подписка, статистика, рекомендации идут на тот же хост и **404** (или падают → mock).  
   Если Next.js: многие роуты есть, но **логин не отдаёт JWT** (только cookie), и авторизация в parent-app ломается. Итог: либо «логин ок, остальное 404/mock», либо «функции есть, авторизация сломана».

3. **Нет реальных броней и оплат для родителей**  
   Роутов `/api/bookings`, `/api/bookings/create-payment-intent`, `/api/bookings/confirm` нет. Родитель не может реально забронировать и оплатить; чекаут — mock. Школе пришлось бы вести брони вне системы или не вести вовсе.

4. **Личность родителя не проверяется при использовании Next.js**  
   Роуты для родителей в Next.js принимают `x-parent-id` (`src/lib/api-auth.ts` 28–36). Сервер **доверяет заголовку**; JWT или подписи нет. Любой клиент может подставить любой `x-parent-id` и получить данные этого родителя.

5. **Нет сброса пароля**  
   Заблокированные родители (забыли пароль) не могут восстановить доступ; поддержке пришлось бы сбрасывать вручную (если для этого вообще есть механизм).

6. **Две БД / две схемы**  
   Next.js Prisma (CUID, School, User, Team, Parent, Player, Feed, Chat, CoachProfile и т.д.) и Prisma в hockey-server (int ID, нет School, другие модели). Нет единого источника правды; общая идентичность «родитель»/«игрок» между CRM и parent-app при использовании обоих не гарантирована.

7. **Авторизация дашборда — только демо**  
   Логин через `DEMO_USERS` и фиксированный пароль «admin123». Нет нормальной таблицы пользователей и RBAC для продакшена; любой, кто знает демо-пароль, получает полный доступ.

8. **Нет rate limiting и защиты от перебора**  
   Логин и API не ограничены по частоте; уязвимы к брутфорсу и злоупотреблениям.

9. **Видео не сохраняется**  
   `POST /api/video/upload` не сохраняет файл; анализ использует stub-хранилище. Любой сценарий «загрузил и проанализировал» — фейк.

10. **Нет продакшн-защиты**  
    Сессионная cookie — base64 JSON (без HMAC/подписи). Нет CSRF, жёсткого CORS, аудит-логов. Для реальных пользователей и соответствия требованиям — небезопасно.

## 5. Минимум работ для безопасного пилота с одной школой

**Цель:** Одна школа, ограниченное число родителей, без оплат пока — но **безопасно** (нет утечек данных, стабильная авторизация, один согласованный бэкенд).

- **Один бэкенд для пилота**  
  Либо:  
  - Направить parent-app на **Next.js** и добавить **выдачу JWT для родителей** в Next.js (в `auth/login` или отдельном роуте входа родителя), использовать этот JWT (и при необходимости `x-parent-id` из него) во всех родительских API; **либо**  
  - Оставить parent-app на **hockey-server** и реализовать **все родительские роуты** там (лента, команда, уведомления, брони только на чтение или заглушка), чтобы в критичных путях не было 404.  
  Рекомендация по скорости: **один бэкенд — Next.js**, JWT для родителей, parent-app перевести на него.

- **Строгая привязка данных к родителю**  
  На каждом эндпоинте, отдающем игроков или данные ребёнка: **фильтр по авторизованному родителю** (например `parentId === req.user.parentId` или из JWT).  
  В hockey-server: в `GET /api/players` и `GET /api/players/:id` ограничить выборку по `parentId === req.user.id`.  
  Если parent-app ходит в Next.js — убрать доверие к голому `x-parent-id`; требовать валидный JWT и брать `parentId` из него.

- **Заменить демо-авторизацию дашборда**  
  Хранить пользователей (и при необходимости школы) в БД; авторизация по БД; оставить текущие `requirePermission` и data-scope по `schoolId`/`teamId`.  
  Для пилота можно обойтись без полного «забыл пароль», если готовы сбрасывать вручную; иначе — минимальный сброс (например ссылка на email + установка нового пароля).

- **Одна БД / одна схема для пилота**  
  Использовать **только** Next.js + корневой Prisma.  
  Либо отключить hockey-server для пилота, либо сделать его тонкой обёрткой над API Next.js, чтобы был один источник правды (одна БД, одна модель Parent/Player/School).

- **Минимальный профиль родителя**  
  Как минимум: родитель может просматривать и редактировать свой профиль (имя, email, смена пароля) через API с проверкой JWT и `parentId`.

- **Что можно не делать в пилоте**  
  Брони можно оставить «скоро» или заглушкой (без реальной оплаты).  
  SMS, верификация email, сохранение видео, подписки — отложить.  
  Важно: везде, где бэкенд ещё заглушка, показывать только нечувствительные или явно помеченные mock-данные.

**Результат:** Один бэкенд (предпочтительно Next.js), JWT для родителей, доступ только к своим данным, реальный логин дашборда, одна БД. Нет IDOR, нет 404 на критичных родительских сценариях, нет демо-пароля в проде.

## 6. Минимум работ для безопасной работы со 100 школами

Всё из раздела 5 плюс:

- **Мультитенантность и масштаб**  
  Все запросы и API **жёстко ограничены по `schoolId`** (и при необходимости `teamId`).  
  Ни один API не должен отдавать данные другой школы (аудит всех `requirePermission` и data-scope; тесты).  
  Индексы и запросы заточены под `schoolId` и размер тенанта; без full-table scan на запрос.

- **Надёжная авторизация и жизненный цикл аккаунта**  
  Восстановление и сброс пароля (ссылка на email или защищённый токен).  
  Верификация email при регистрации (или хотя бы перед чувствительными действиями).  
  Инвалидация сессии (logout реально инвалидирует токен/сессию на сервере); срок жизни JWT и refresh при использовании JWT.

- **Реальные брони и оплаты (если школы берут деньги)**  
  Реализовать `/api/bookings` (создание, чтение, подтверждение, отмена) с привязкой к школе/тренеру.  
  Интеграция Stripe (или одного платёжного провайдера) для оплат родителями; webhook для обновления статуса брони/оплаты; идемпотентность и базовая сверка.

- **Защита от злоупотреблений**  
  Rate limiting на логин и чувствительные API (по IP и по пользователю).  
  Подписанная или серверная сессия (не голый base64 в cookie); только HTTPS; безопасные флаги cookie.  
  Никакого доверия к идентификатору с клиента (например, не принимать `x-parent-id` без проверенного JWT).

- **Готовность к эксплуатации**  
  Health checks и мониторинг; трекинг ошибок (например Sentry).  
  Логирование (кто что сделал, какая школа); без логирования паролей и токенов.  
  Бэкапы и восстановление единой БД; документированный runbook при инцидентах.

- **Поддержка и соответствие**  
  Возможность для поддержки «войти как» родитель (с аудитом) или сбросить пароль.  
  Базовый экспорт данных (например родители/игроки по школе) для запросов по GDPR/школе.  
  Ясные политики конфиденциальности и условия использования, ссылки в приложении при необходимости.

- **Производительность и лимиты**  
  Пагинация на всех списковых API (игроки, брони, лента и т.д.).  
  Нет N+1 запросов; пул соединений; при росте нагрузки — рассмотреть read replica.  
  Задать и проверять лимиты (макс. игроков на школу, макс. броней на родителя и т.д.), чтобы одна школа не перегружала систему.

**Результат:** Безопасная мультитенантность, реальный жизненный цикл авторизации, оплаты/брони при необходимости, rate limiting, безопасные сессии, мониторинг, бэкапы и пути поддержки/экспорта. Нет утечек между школами; система выдерживает 100 школ и рост.
