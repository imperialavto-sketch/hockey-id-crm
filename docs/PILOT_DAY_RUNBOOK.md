# Pilot Day Runbook — Hockey ID (controlled pilot)

Короткая инструкция **на день** controlled pilot. Выполнять по порядку; при срабатывании stop conditions — остановка и разбор, не «дожимать» демо.

## 1. Цель runbook

Подтвердить на **одной связке** (один API + одна БД + одни учётные данные пилота):

- сервер Next и миграции БД;
- автоматический smoke по HTTP;
- ручные проверки **CRM web**, **coach-app**, **parent-app**.

Если любой блок падает — пилот в этой конфигурации не подтверждён.

## 2. Перед стартом (5–10 мин)

| Шаг | Действие |
|-----|----------|
| Git | Зафиксировать **commit** и **branch** (запись в чат/тикет пилота). |
| БД | **`DATABASE_URL`** — тот инстанс PostgreSQL, куда смотрит **этот** деплой Next. |
| Smoke / curl | **`ORIGIN`** — базовый URL API без завершающего `/` (локально или `https://<pilot-host>`). |
| Мобилки | **`EXPO_PUBLIC_API_URL`** на устройстве/сборке = **тот же хост**, что пилот (не `localhost` на физическом телефоне без туннеля). |
| Режим | **`EXPO_PUBLIC_DEMO_MODE=false`**. |
| Fallback | **`EXPO_PUBLIC_ENABLE_API_FALLBACK=false`** — или явно в тикете: «включён fallback, риск скрытых ошибок API». |
| Среда | **`NODE_ENV`** как ожидается для стенда (обычно `production` на боевом хосте; от этого зависят 410 на demo/mock — см. код сервера). |

Учётные данные пилота заранее в `docs/PILOT_SMOKE.md` / сид `db:seed:pilot`.

## 3. Server / DB — команды (выполнить по порядку)

На машине/CI, где крутится пилотский Next (или перед деплоем):

```bash
npx prisma migrate deploy
npm run build
```

После деплоя Next (сервер слушает пилотский URL):

```bash
npm run db:seed:pilot
```

Автоматический smoke **с той же машины, где виден API**, или с VPN:

```bash
# локально
npm run smoke:pilot:live

# удалённый пилотский хост
ORIGIN=https://<pilot-host> npm run smoke:pilot:live
```

При ошибке — **стоп**, раздел 8 и 11.

## 5. CRM web — ручной чеклист

В браузере на пилотском URL:

- [ ] Логин (учётка по политике пилота — часто тот же coach или admin школы).
- [ ] Открывается dashboard.
- [ ] Teams — список / нужная команда.
- [ ] Players — список, пилотный игрок присутствует при ожидании.
- [ ] Тренировка / слот / отчёт — видимость после publish (нет «пустого» критичного экрана).
- [ ] DevTools → Network: **нет 5xx** на основных переходах; Console без критичных ошибок.

## 6. Coach app — ручной чеклист

Сборка с пилотским `.env` (`EXPO_PUBLIC_*` из раздела 2):

- [ ] Логин **pilot-coach** (email/пароль выше).
- [ ] Расписание грузится.
- [ ] Пилотный слот виден (неделя с **`WEEK_START`** по умолчанию **2026-04-20** — см. `docs/PILOT_SMOKE.md`).
- [ ] Открыть слот / тренировку.
- [ ] Старт live training.
- [ ] Finish.
- [ ] Confirm при необходимости (если не auto-confirmed).
- [ ] Report draft открывается.
- [ ] Publish успешен.
- [ ] Назад по навигации — экраны не «ломаются» (нет белого экрана / вечного лоадера).

## 7. Parent app — ручной чеклист

- [ ] Логин **pilot-parent**.
- [ ] Игрок **PilotSmoke Player** виден в списке.
- [ ] Профиль игрока открывается.
- [ ] Блок «последняя тренировка» / summary **виден**.
- [ ] Текст **не пустой** (осмысленные строки после publish).
- [ ] Убедиться, что приложение **не** в demo-режиме и не маскирует ошибки API (раздел 2).

Допустимо: в ответе API **`source: live_session_fallback`** — это не провал runbook само по себе, если данные не пустые (см. `docs/PILOT_SMOKE.md`).

## 8. Stop conditions — пилот останавливается

Немедленно остановить пилотский прогон и не называть день успешным, если:

- падает **`migrate deploy`**;
- падает **`npm run build`**;
- падает **`smoke:pilot:live`**;
- мобильное приложение **не достучалось** до API (таймаут, SSL, неверный host);
- coach **не может** publish (4xx/5xx или застревание UI);
- parent **не видит** summary или текст пустой при успешном publish у coach;
- **5xx** на главном пути (login, schedule, live, publish, parent summary);
- **auth** ведёт себя непредсказимо (чужие данные, пустая сессия, обход без пароля).

Дальше — логи, тикет, раздел 11.

## 9. Evidence (что сохранить в тикет)

| Артефакт | Зачем |
|----------|--------|
| Полный вывод терминала **`smoke:pilot:live`** | Доказательство автоматической цепочки. |
| Скрин или короткое видео **coach → publish success** | Ручной продуктовый контур. |
| Скрин **parent → latest summary** | Конец value chain. |
| Фрагмент **server logs** вокруг времени publish | Корреляция 5xx/Prisma. |
| **Env snapshot без секретов** (имена ключей + не-чувствительные значения) | Воспроизводимость. |

## 10. Известные допустимые ограничения

- **`source: live_session_fallback`** в parent summary — **допустимо** для этого пилота, если `hasData` и текст есть.
- **`OPENAI_API_KEY`** для описанного smoke **не обязателен**.
- **Push, чат, полный Arena, все экраны** — этим runbook **не покрываются**.
- **Controlled pilot ≠** полная production readiness.

## 11. Emergency rollback / reset

1. Остановить демонстрацию / доступ пилотским пользователям при необходимости (коммуникация).  
2. Собрать **логи** Next + БД за окно инцидента.  
3. Повторить **`npx prisma migrate deploy`** если подозрение на рассинхрон схемы.  
4. При чистом повторе сценария на **той же** пилотской БД: **`npm run db:seed:pilot`** (идемпотентно).  
5. Если **`smoke:pilot:live`** даёт **409** «активная live-сессия» — в UI/API **завершить или отменить только пилотскую** live-сессию того же coach; **не удалять** чужие команды/игроков/школы.  
6. Не выполнять **`prisma db push --force-reset`** и не чистить БД целиком без явного решения владельца данных.
