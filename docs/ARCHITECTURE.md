# Hockey ID — Архитектурный план экосистемы

## Обзор

Hockey ID — экосистема из двух самостоятельных продуктов с общей бизнес-логикой и базой данных:

| Продукт | Назначение | Платформа | Пользователи |
|---------|------------|-----------|--------------|
| **hockey-id-crm** | Внутренняя CRM школы | Web (Next.js) | Администратор, тренеры, менеджер |
| **hockey-id-parent-app** | Приложение для родителей | Mobile (iOS, Android) | Родители |

---

## 1. Продукт 1: CRM школы (hockey-id-crm)

### Назначение
Внутренняя web-система для управления хоккейной школой: команды, игроки, тренеры, расписание, финансы, аналитика.

### Пользователи
- **Администратор школы** — полный доступ
- **Главный тренер** — управление командами, тренерами, расписанием
- **Тренер** — свои команды, тренировки, оценки игроков
- **Менеджер школы** — игроки, расписание, коммуникации, финансы

### Основные модули

| Модуль | Описание |
|--------|----------|
| Dashboard | KPI, ближайшие тренировки, активность, сводная аналитика |
| Команды | CRUD команд, игроки команды |
| Игроки | Список игроков, карточка игрока, паспорт, статистика |
| Тренеры | Тренеры, оценки, рекомендации, журнал тренировок |
| Расписание | Тренировки, посещаемость |
| Финансы | Оплаты, задолженности, массовое начисление |
| Коммуникации | Сообщения родителям, уведомления |
| Аналитика | Игроки, посещаемость, финансы, тренеры |
| Настройки | Школа, пользователи, роли, права, интеграции |

### Технологии
- Next.js 14, React, TailwindCSS
- Prisma, PostgreSQL

## 3. Общая архитектура данных

### Схема репозиториев

```
hockey-id/
├── hockey-id-crm/          # Web CRM
├── hockey-id-parent-app/   # Mobile приложение
├── hockey-id-api/          # [Опционально] общий API
└── hockey-id-shared/       # [Опционально] shared-логика
```

### Варианты архитектуры API

**Вариант A: API внутри каждого продукта (текущий)**
- CRM: `hockey-id-crm/src/app/api/`
- Parent App: API в backend-части Expo или отдельный backend

**Вариант B: Общий API (рекомендуется для масштабирования)**
- Выделенный `hockey-id-api` (Next.js API Routes или отдельный Node/Express)
- Оба клиента обращаются к одному API
- Единая точка валидации, авторизации и бизнес-логики

### Основные сущности (общая Prisma schema)

| Сущность | Описание | CRM | Parent App |
|----------|----------|-----|------------|
| School | Школа | ✓ полный доступ | — |
| Team | Команда | ✓ полный доступ | ✓ только команда ребёнка |
| Player | Игрок | ✓ все игроки | ✓ только свои дети |
| Coach | Тренер | ✓ все тренеры | ✓ тренеры команды ребёнка |
| Training | Тренировка | ✓ все | ✓ тренировки ребёнка |
| Attendance | Посещаемость | ✓ все | ✓ посещаемость ребёнка |
| PlayerPayment | Оплата | ✓ все | ✓ только свои платежи |
| PlayerProfile | Профиль | ✓ | ✓ |
| Passport | Паспорт | ✓ | ✓ |
| Skills | Навыки | ✓ | ✓ |
| Medical | Медкарта | ✓ | ✓ |
| Achievement | Достижение | ✓ | ✓ |
| Video | Видео | ✓ | ✓ |
| CoachRating | Оценка тренера | ✓ | ✓ |
| Parent | Родитель | ✓ | ✓ свой профиль |
| Message | Сообщение | ✓ все | ✓ свои |
| User | Пользователь | ✓ школы | ✓ родители (логин) |
| Role, Permission | Роли и права | ✓ | — |

## 4. Разделение доступа по ролям

### Данные для CRM (школа)
- Полный доступ: School, Team, Player, Coach, Training, Attendance, Payment, и т.д.
- Создание, редактирование и удаление всех сущностей в рамках своей школы.

### Данные для Parent App
- **Только чтение** (кроме своих сообщений/профиля):
  - Профиль своего ребёнка (Player + PlayerProfile + Passport + Skills + Medical + Achievement + Video)
  - Расписание тренировок ребёнка (Training + Attendance)
  - Оценки и рекомендации тренера (CoachRating)
  - Оплаты по своему ребёнку (PlayerPayment)
- **Чтение и запись**:
  - Сообщения (Message) — свои
  - Профиль родителя (Parent) — свой

### Карта API endpoints

| Endpoint | CRM | Parent App | Описание |
|----------|-----|------------|----------|
| `/api/schools` | ✓ | ✗ | Школы |
| `/api/teams` | ✓ | ✗ | Все команды (CRM) |
| `/api/teams/[id]` | ✓ | ✓* | Команда (*только команда ребёнка) |
| `/api/players` | ✓ | ✗ | Все игроки (CRM) |
| `/api/players/[id]` | ✓ | ✓* | Игрок (*только свой ребёнок) |
| `/api/coaches` | ✓ | ✓* | Тренеры (*команды ребёнка) |
| `/api/trainings` | ✓ | ✗ | Все тренировки (CRM) |
| `/api/trainings/my-child` | ✗ | ✓ | Тренировки ребёнка |
| `/api/attendance` | ✓ | ✓* | Посещаемость (*ребёнка) |
| `/api/payments` | ✓ | ✗ | Все платежи (CRM) |
| `/api/payments/my` | ✗ | ✓ | Платежи родителя |
| `/api/ratings` | ✓ | ✓* | Оценки (*ребёнка) |
| `/api/messages` | ✓ | ✓* | Сообщения (*свои) |
| `/api/auth/login` | ✓ | ✓ | Общий вход |
| `/api/dashboard/*` | ✓ | ✗ | Dashboard школы |
| `/api/settings/*` | ✓ | ✗ | Настройки школы |
| `/api/analytics/*` | ✓ | ✗ | Аналитика |

## 6. Auth и роли

### CRM
- Роли: `SCHOOL_ADMIN`, `MAIN_COACH`, `COACH`, `SCHOOL_MANAGER`
- JWT/Session с `role` и `schoolId`
- Middleware проверяет роль и schoolId

### Parent App
- Роль: `PARENT`
- JWT/Session с `parentId` и привязкой к `children: [playerId, ...]`
- Каждый запрос проверяет, что данные относятся к ребёнку родителя

### Общие принципы
- Один `User` для входа
- Поле `role` определяет, к какому продукту относится пользователь
- Для PARENT — связь `User` → `Parent` → `Player` (children)

## 9. Рекомендации

### Что оставить в hockey-id-crm
- Всё управление школой: команды, игроки, тренеры, расписание
- Финансы, аналитика, настройки
- Роли и права доступа
- Полный CRUD по сущностям
- Отправка сообщений родителям (односторонняя коммуникация из CRM)

### Что вынести в hockey-id-parent-app
- Просмотр данных ребёнка
- Просмотр расписания и посещаемости
- Просмотр оплат и оплата
- Чат с тренером
- Push-уведомления
- Любой UI, адресованный родителю

### Общий API
- Использовать единый базовый URL (или gateway)
- Всегда передавать `Authorization` и проверять роль
- Для Parent App — фильтровать данные по `parentId` → `playerId`

### Auth
- Один User на email
- `role = PARENT` → показывать родительское приложение
- `role ∈ [SCHOOL_ADMIN, MAIN_COACH, COACH, SCHOOL_MANAGER]` → CRM
- Разные приложения для входа: CRM — браузер, Parent — мобильное приложение
